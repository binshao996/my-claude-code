# 从 0 到 1 实现 Claude Code：V0.1 工程骨架、核心协议和 DeepSeek Spike

## 这一章要做什么

V0.1 不是为了做出一个完整 agent，而是为了把以后不会轻易推倒的“地基”搭好。Claude Code 这类产品不是简单的“命令行 + LLM API”，它后面会逐步长出：

- CLI 和 TUI。
- 可流式输出、可中断、可恢复的 agent loop。
- 工具调用和权限系统。
- transcript、session、resume。
- MCP、skills、plugins。
- subagent、background task、remote。

如果 V0.1 一开始就直接写“用户输入 -> 调模型 -> 输出文本”，后面做到工具、权限、上下文压缩时会反复推翻。所以本章只做 5 件事：

1. 初始化 Bun + TypeScript monorepo。
2. 定义 Claude-compatible 内部协议。
3. 把 DeepSeek 的 streaming 响应适配成内部协议事件。
4. 做一个最小 CLI，能跑 `--help`、`--version`、DeepSeek compatibility spike。
5. 支持开发阶段 `.env`，生产阶段系统环境变量。

完成本章后，你还不能让 agent 修改代码，但你已经有了未来实现 agent loop、tool loop、session replay 的基础协议。

## 两种实现方式

你可以用“命令 + 手写代码”的方式实现，也可以把 prompt 交给 AI 实现。建议实际学习时两种都看：命令帮你理解工程初始化，prompt 帮你训练如何把需求交给 coding agent。

### 方式 A：命令初始化

从空目录开始：

```sh
mkdir my-claude-code
cd my-claude-code

bun init -y
mkdir -p packages/core/src
mkdir -p packages/model-provider/src
mkdir -p packages/cli/src
mkdir -p tech-docs

bun add zod @commander-js/extra-typings
bun add -d typescript @types/bun @types/node @biomejs/biome
```

然后手动创建这些根文件：

- `package.json`
- `tsconfig.json`
- `biome.json`
- `.gitignore`
- `.env.example`

根 `package.json` 的核心 scripts 应该是：

```json
{
  "scripts": {
    "build": "bun build packages/cli/src/cli.ts --target=node --outfile=dist/cli.js && chmod +x dist/cli.js",
    "cli": "bun run packages/cli/src/cli.ts",
    "dev": "bun run packages/cli/src/cli.ts",
    "lint": "biome lint packages docs tech-docs README.md package.json tsconfig.json biome.json",
    "test": "bun test ./packages/core/src ./packages/model-provider/src ./packages/cli/src",
    "typecheck": "tsc --noEmit"
  }
}
```

这里有一个容易踩的坑：仓库里有 `claude-code/` 参考源码，所以 `test` 和 `lint` 必须限定目录。如果写成 `bun test` 或 `biome lint .`，会把参考源码测试也跑进去，结果既慢又会误报。

### 方式 B：交给 AI 的 Prompt

如果你想让 AI 直接生成 V0.1，可以使用这个 prompt：

```text
你是资深 TypeScript/Bun 工程师。我要从 0 到 1 实现一个 Claude Code-like agent 的 V0.1。

目标：
1. 建立 Bun + TypeScript strict monorepo。
2. 包边界为 packages/core、packages/model-provider、packages/cli。
3. core 定义 Claude-compatible 协议：TextBlock、ThinkingBlock、ToolUseBlock、ToolResultBlock、ContentBlock、Message、Usage、StopReason、QueryEvent、TranscriptRecord。
4. model-provider 实现 DeepSeek/OpenAI-compatible streaming chunk 到 QueryEvent 的 adapter，支持 text delta、reasoning_content/thinking_delta、tool_call arguments/input_json_delta、usage、finish_reason、SSE [DONE]。
5. cli 使用 @commander-js/extra-typings，实现 --version、--help、--compatibility-spike、--compatibility-spike-live。
6. 开发环境支持读取 .env/.env.local，NODE_ENV=production 时不读取 .env，只读系统环境变量。
7. 不要把任何真实 API key 写进代码或文档。
8. 添加 bun test 单测，覆盖协议 schema、DeepSeek adapter、CLI fast path、.env loader。
9. scripts 必须包含 typecheck、test、lint、build。

约束：
- 使用 zod/v4 做 runtime schema。
- 内部协议保持 Claude-compatible，不要让 query loop 依赖 DeepSeek 原始 chunk。
- lint/test 只扫描新实现目录，不要扫描 claude-code/ 参考源码。
- 使用小而清晰的文件，不要一次性写巨型文件。

验收：
- bun run typecheck 通过。
- bun run test 通过。
- bun run lint 通过。
- bun run build 通过。
- bun run cli -- --help 能输出 usage。
- bun run cli -- --compatibility-spike-live 能从 DEEPSEEK_API_KEY 读取 key，但不得打印 key。
```

这个 prompt 的重点是把“边界”和“验收”说清楚。否则 AI 很容易直接写一个单文件 CLI，看起来能跑，但后面无法扩展成 Claude Code。

## Step 1：初始化 Monorepo

### 为什么要 Monorepo

Claude Code 的源码不是一个单包 CLI，它有多个 package：

- CLI 主程序。
- terminal renderer。
- provider abstraction。
- builtin tools。
- MCP client。
- native packages。

V0.1 不需要复刻全部 package，但要先建立 package 边界。否则后面 provider、tools、CLI、TUI 全堆在一个 `src/` 里，会很快失控。

### V0.1 的三个包

| Package | 职责 | 不应该做什么 |
| --- | --- | --- |
| `@my-claude-code/core` | 定义内部协议、基础 feature flag | 不调用模型、不读文件、不做 CLI |
| `@my-claude-code/model-provider` | 把 provider 原始请求/响应转成内部协议 | 不处理 CLI 参数、不做工具执行 |
| `@my-claude-code/cli` | 命令行入口、参数解析、开发环境加载 | 不直接解析 DeepSeek chunk，不定义核心协议 |

这三个包对应一种重要设计原则：外部世界可以变，但内部协议要稳定。

DeepSeek API、OpenAI API、未来其他 provider 都是外部世界。CLI/TUI、agent loop、tools 是内部 runtime。`model-provider` 的工作就是把外部 provider 的差异抹平，让内部 runtime 只看统一的 `QueryEvent`。

## Step 2：定义核心协议

### 协议是什么

这里的“协议”不是网络协议，而是 agent runtime 内部传递消息的标准格式。

后续 agent loop 会不断处理 provider 流式返回的事件。所谓 agent loop，可以先理解成一个循环：

1. 读取用户输入。
2. 把用户输入和历史上下文发给模型。
3. 一边接收模型的 streaming 事件，一边更新 UI/transcript。
4. 如果模型要求调用工具，就暂停模型输出，执行工具。
5. 把工具结果再发回模型。
6. 重复这个过程，直到模型以普通文本结束。

简化流程是：

```text
用户输入
  -> provider streaming
  -> message_start
  -> content_block_start
  -> content_block_delta
  -> content_block_stop
  -> message_delta
  -> message_stop
  -> 写 transcript
  -> 如果有 tool_use，执行工具，再继续下一轮
```

举一个具体例子。用户输入：

```text
查看当前目录
```

模型可能不是直接回答，而是先请求调用 `Bash` 工具：

```text
message_start
content_block_start: tool_use(name="Bash", id="toolu_1")
content_block_delta: input_json_delta('{"command"')
content_block_delta: input_json_delta(':"pwd"}')
content_block_stop
message_delta: stop_reason="tool_use"
message_stop
```

agent loop 看到这些事件后会做几件事：

| 事件 | agent loop 做什么 |
| --- | --- |
| `message_start` | 创建一个新的 assistant message |
| `content_block_start(tool_use)` | 知道模型要调用工具，记录工具名和工具调用 ID |
| `input_json_delta` | 拼接工具参数 JSON，例如最终得到 `{"command":"pwd"}` |
| `content_block_stop` | 标记这个工具调用参数接收完毕 |
| `message_delta.stop_reason = "tool_use"` | 暂停模型回合，进入工具执行阶段 |
| `message_stop` | 当前 assistant streaming 结束 |

然后 agent loop 会执行工具，得到结果：

```text
/Users/me/project
```

再把工具结果包装成下一轮用户侧消息：

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_1",
      "content": "/Users/me/project"
    }
  ]
}
```

模型收到工具结果后继续输出最终回答：

```text
message_start
content_block_start: text
content_block_delta: "当前目录是 /Users/me/project"
content_block_stop
message_delta: stop_reason="end_turn"
message_stop
```

这就是“agent loop 不断处理事件”的意思：它不是等模型一次性返回完整文本，而是在 streaming 过程中根据事件类型实时决定下一步做什么。

如果没有统一协议，会发生什么？

| 没有协议的写法 | 后果 |
| --- | --- |
| CLI 直接解析 DeepSeek 的 `delta.tool_calls` | 以后换 provider，CLI 要改 |
| TUI 直接解析 DeepSeek 的 `delta.content` | 以后 Claude/Gemini/OpenAI 字段不同，TUI 要改 |
| transcript 直接保存 provider 原始 chunk | resume 时必须理解所有 provider 的历史格式 |
| tool loop 直接读 provider 原始 JSON | 工具调用、权限、错误恢复会和 provider 强绑定 |
| 没有统一 `stop_reason` | 不知道什么时候该结束、什么时候该执行工具、什么时候该继续请求 |

统一协议的价值就是：provider 怎么返回是外部问题，agent runtime 只处理自己的 `QueryEvent`。

### 为什么是 Claude-compatible

目标是 1:1 复刻 Claude Code，所以内部协议要尽量贴近 Claude 的事件模型，而不是贴 DeepSeek：

“Claude 的事件模型”可以理解成 Anthropic/Claude 在 streaming 时使用的一种分层结构：

```text
Message
└── ContentBlock 0
    ├── start
    ├── delta
    ├── delta
    └── stop
└── ContentBlock 1
    ├── start
    ├── delta
    └── stop
└── Message delta / stop
```

它不是只输出一段字符串，而是把一次 assistant 回复拆成：

- message 级别：这是一条 assistant message。
- block 级别：这段内容是 text、thinking、tool_use 还是 tool_result。
- delta 级别：streaming 过程中每次增量输出了什么。
- stop 级别：模型为什么停下来。

为什么这个模型适合 Claude Code？

| 能力 | 为什么需要事件模型 |
| --- | --- |
| 流式显示文本 | `text_delta` 来一段显示一段 |
| 工具调用 | `tool_use` block 可以和普通文本区分开 |
| 工具参数增量生成 | `input_json_delta` 可以一点点拼完整 JSON |
| thinking/reasoning | `thinking` block 可以保存推理内容，后续多轮回传 |
| transcript replay | message/block/delta 结构可以还原一次会话 |
| TUI 渲染 | 不同 block 可以渲染成不同 UI |
| 权限系统 | 看到 `tool_use` 后先走权限判断，再决定是否执行 |

DeepSeek 的 API 是 OpenAI-compatible，但我们的目标不是复刻 OpenAI CLI，而是复刻 Claude Code。所以内部事件模型要贴近 Claude Code；DeepSeek 只是被 adapter 转成这种模型。

| 内部类型 | 用途 |
| --- | --- |
| `TextBlock` | assistant 普通文本输出 |
| `ThinkingBlock` | reasoning/thinking 内容，DeepSeek 的 `reasoning_content` 会映射到这里 |
| `ToolUseBlock` | 模型请求调用工具 |
| `ToolResultBlock` | 工具执行结果返回给模型 |
| `ContentBlock` | 上面几种 block 的联合类型 |
| `Message` | user/assistant/system 消息 |
| `Usage` | token 用量 |
| `StopReason` | 模型停止原因，如 `end_turn`、`tool_use`、`max_tokens` |
| `QueryEvent` | streaming 事件，是后续 agent loop 的核心输入 |
| `TranscriptRecord` | 写入 JSONL transcript 的记录 |

### 为什么用 Zod

TypeScript 类型只在编译期有效，但 provider 返回的数据来自网络，运行时可能缺字段、字段类型不对、JSON 不完整。Zod 的作用是：

- 测试里验证事件结构。
- 后续 transcript 写入前做校验。
- provider adapter 出错时能尽早暴露。
- CLI/SDK 未来输出 JSON 时有稳定 schema。

举个例子。一个合法的 `tool_use` block 应该长这样：

```ts
ToolUseBlockSchema.parse({
  type: 'tool_use',
  id: 'toolu_1',
  name: 'Bash',
  input: {
    command: 'pwd',
  },
})
```

如果 adapter 误把 `input` 输出成字符串：

```ts
ToolUseBlockSchema.parse({
  type: 'tool_use',
  id: 'toolu_1',
  name: 'Bash',
  input: '{"command":"pwd"}',
})
```

Zod 会在运行时报错，因为 `input` 必须是 object。这个错误应该在 V0.1 的 parser/test 阶段暴露，而不是等到 V0.3 真正执行工具时才发现。

再看 streaming event：

```ts
QueryEventSchema.parse({
  type: 'content_block_delta',
  index: 0,
  delta: {
    type: 'input_json_delta',
    partial_json: '{"command"',
  },
})
```

如果有人写成：

```ts
QueryEventSchema.parse({
  type: 'content_block_delta',
  index: 0,
  delta: {
    type: 'tool_json_delta',
    json: '{"command"',
  },
})
```

这在 TypeScript 编译时可能因为 `unknown` 或外部 JSON 混过去，但 Zod 会拒绝。这样 transcript、TUI、agent loop 就不需要处理一堆“也许存在”的字段。

### 这一版要实现什么

在 `packages/core/src/protocol.ts` 中实现：

- `TextBlockSchema`
- `ThinkingBlockSchema`
- `ToolUseBlockSchema`
- `ToolResultBlockSchema`
- `ContentBlockSchema`
- `MessageSchema`
- `UsageSchema`
- `StopReasonSchema`
- `QueryEventSchema`
- `TranscriptRecordSchema`

同时导出对应 TypeScript 类型。

单测至少覆盖：

- tool use block 可以通过校验。
- text streaming delta 可以通过校验。
- thinking block 和 thinking delta 可以通过校验。
- transcript record 可以通过校验。

## Step 3：实现 DeepSeek Stream Adapter

### 什么是 OpenAI-Compatible Chat Completion Streaming

DeepSeek 的 `/chat/completions` 接口使用 OpenAI-compatible 格式。意思是它的请求/响应形状大体接近 OpenAI Chat Completions：

请求大致是：

```json
{
  "model": "deepseek-v4-flash",
  "messages": [{ "role": "user", "content": "..." }],
  "tools": [],
  "stream": true
}
```

streaming 响应不是一次性 JSON，而是 SSE 形式的一串 `data:` 行：

```text
data: {"id":"...","choices":[{"delta":{"content":"hello"},"finish_reason":null}]}
data: {"choices":[{"delta":{"tool_calls":[...]}}]}
data: {"choices":[{"delta":{},"finish_reason":"tool_calls"}],"usage":{...}}
data: [DONE]
```

所以 adapter 要做两层转换：

1. SSE line -> DeepSeek/OpenAI chunk。
2. DeepSeek/OpenAI chunk -> Claude-compatible `QueryEvent`。

把这两层拆开看会更清楚。

第一层：SSE line -> DeepSeek/OpenAI chunk。

DeepSeek streaming 返回的是文本流，每一行大概这样：

```text
data: {"id":"chatcmpl_1","choices":[{"index":0,"delta":{"content":"hello"},"finish_reason":null}]}
```

这一层只做“拆包”：

```text
去掉 data:
  -> 得到 JSON 字符串
JSON.parse
  -> 得到 DeepSeekChatCompletionChunk
```

拆完以后得到的对象类似：

```json
{
  "id": "chatcmpl_1",
  "choices": [
    {
      "index": 0,
      "delta": {
        "content": "hello"
      },
      "finish_reason": null
    }
  ]
}
```

第二层：DeepSeek/OpenAI chunk -> Claude-compatible `QueryEvent`。

上面的 chunk 会被转换成内部事件：

```json
[
  {
    "type": "message_start",
    "message": {
      "id": "chatcmpl_1",
      "role": "assistant",
      "model": "deepseek-v4-flash"
    }
  },
  {
    "type": "content_block_start",
    "index": 0,
    "content_block": {
      "type": "text",
      "text": ""
    }
  },
  {
    "type": "content_block_delta",
    "index": 0,
    "delta": {
      "type": "text_delta",
      "text": "hello"
    }
  }
]
```

再举一个 tool call 的例子。DeepSeek 可能分两次返回工具参数：

```text
data: {"choices":[{"delta":{"tool_calls":[{"index":0,"id":"call_1","type":"function","function":{"name":"Bash","arguments":"{\"command\""}}]},"finish_reason":null}]}
data: {"choices":[{"delta":{"tool_calls":[{"index":0,"function":{"arguments":":\"pwd\"}"}}]},"finish_reason":"tool_calls"}]}
```

adapter 要把它变成：

```text
content_block_start: tool_use(id="call_1", name="Bash", input={})
content_block_delta: input_json_delta('{"command"')
content_block_delta: input_json_delta(':"pwd"}')
content_block_stop
message_delta: stop_reason="tool_use"
message_stop
```

注意：adapter 在这里不执行工具，也不解析最终 JSON 是否安全。它只负责把 streaming 结构转换成内部协议。真正的工具参数解析、权限判断、执行工具，是 V0.3 以后 tool loop 的职责。

为什么不把两层合在一起？因为它们解决的问题不同：

| 层级 | 解决的问题 |
| --- | --- |
| SSE parser | 网络流拆包、处理 `data:`、处理 `[DONE]` |
| chunk adapter | 理解 provider 字段，把 content/tool/reasoning/usage 映射成内部事件 |

这样以后如果 provider 不是 SSE，而是 WebSocket，只需要换第一层；如果 provider 字段不同，只需要换第二层。

### 为什么参考 OpenAI-compatible 而不是 Claude API

项目目标是复刻 Claude Code 的行为，但底层 provider 指定为 `deepseek-v4-flash`。DeepSeek 的 API 是 OpenAI-compatible，不是 Anthropic Claude API。

因此正确做法是：

- 内部协议参考 Claude，因为要复刻 Claude Code。
- 外部输入解析参考 OpenAI-compatible，因为 DeepSeek 按这个格式返回。
- 中间用 adapter 把二者接起来。

### 字段映射

| DeepSeek/OpenAI chunk | 内部 QueryEvent |
| --- | --- |
| 第一条 chunk | `message_start` |
| `delta.reasoning_content` | `content_block_start(thinking)` + `thinking_delta` |
| `delta.content` | `content_block_start(text)` + `text_delta` |
| `delta.tool_calls[].function.arguments` | `content_block_start(tool_use)` + `input_json_delta` |
| `finish_reason: "stop"` | `message_delta.stop_reason = "end_turn"` |
| `finish_reason: "tool_calls"` | `message_delta.stop_reason = "tool_use"` |
| `finish_reason: "length"` | `message_delta.stop_reason = "max_tokens"` |
| `usage.prompt_tokens` | `Usage.input_tokens` |
| `usage.completion_tokens` | `Usage.output_tokens` |
| `data: [DONE]` | close open blocks + `message_stop` |

### DeepSeek V4 Thinking Mode 的特殊点

DeepSeek V4 thinking mode 指的是：模型在输出最终答案前，会先生成一段 reasoning，也就是它内部的“思考内容”。在 API streaming 里，这段内容不是放在 `delta.content`，而是放在 `delta.reasoning_content`。

一个简化的流可能是：

```text
data: {"choices":[{"delta":{"reasoning_content":"I need to inspect the request."},"finish_reason":null}]}
data: {"choices":[{"delta":{"content":"答案是 ..."},"finish_reason":null}]}
data: {"choices":[{"delta":{},"finish_reason":"stop"}]}
```

adapter 要把它转成两个 block：

```text
content_block_start: thinking
content_block_delta: thinking_delta("I need to inspect the request.")
content_block_stop
content_block_start: text
content_block_delta: text_delta("答案是 ...")
content_block_stop
message_delta: stop_reason="end_turn"
message_stop
```

为什么这件事对 Claude Code-like agent 很重要？

普通聊天里，你可以不关心 reasoning，只显示最终答案。但 agent 做工具调用时不同。DeepSeek V4 thinking mode 如果在一次 assistant turn 里产生了 tool call，后续请求需要把这轮的 reasoning content 保留下来并回传。否则多轮工具调用时可能 400。

比如：

```text
Turn 1:
用户：查看目录
模型 reasoning_content：我需要调用 Bash。
模型 tool_calls：Bash({"command":"pwd"})

工具结果：/Users/me/project

Turn 2:
把工具结果发回模型，让模型继续回答
```

如果 Turn 2 只回传 tool call 和 tool result，不保留 Turn 1 的 reasoning content，DeepSeek thinking mode 可能认为上下文不完整。对应到我们的内部协议，就是必须把 `reasoning_content` 保存成 `ThinkingBlock`，这样 transcript 和后续请求构造器都能拿到它。

实测和官方文档都说明 DeepSeek V4 thinking mode 有两个关键兼容点：

- 不要强制发送 `tool_choice`。thinking mode 可能返回 `Thinking mode does not support this tool_choice`。
- 如果一次 assistant turn 里出现了 tool call，后续多轮请求必须保留 reasoning content。内部协议里要把 `reasoning_content` 保存为 `thinking` block。

这就是为什么 V0.1 必须支持 `ThinkingBlock`。如果你跳过它，V0.2/V0.3 做工具循环时会遇到 400。

### Live Spike 做什么

`--compatibility-spike-live` 做两个真实请求：

1. text probe：验证 text streaming、thinking streaming、usage、stop reason。
2. tool probe：提供 `probe_echo` tool，让模型自己触发 tool call，验证 tool-call argument delta。

它不会执行任何真实工具，也不会写文件。它只验证 DeepSeek 是否能产生 Claude Code 后续需要的 streaming 信号。

成功输出应该包含：

- `status: "passed"`
- `probes.thinkingStreaming: true`
- `probes.textStreaming: true`
- `probes.toolCallDelta: true`
- `probes.usageMapping: true`
- `probes.stopReasonMapping: true`
- `requiresPromptWrappedToolCalls: false`
- `requiresReasoningContentForToolCalls: true`
- `errors: []`

### 本次 V0.1 Live Spike 结论

2026-05-22 使用 `deepseek-v4-flash` 和官方 `https://api.deepseek.com/chat/completions` endpoint 完成 live spike，结果为 `passed`。

结论：

- text streaming 可用。
- thinking streaming 可用。
- tool call argument delta 可用。

- stop reason mapping 可用。

## Step 5：实现开发环境 `.env`

### 为什么要 `.env`

开发阶段每次手动 export key 很麻烦。正式环境又不能把 key 写在代码里。折中方案：

- 开发阶段：自动读 `.env` 和 `.env.local`。
- 生产阶段：`NODE_ENV=production` 时不读 `.env`，只读系统环境变量。

这和 Claude Code 类产品的配置思路一致：本地开发可以方便，但正式运行必须依赖外部安全配置。

### 规则

| 场景 | 行为 |
| --- | --- |
| `NODE_ENV=production` | 不读 `.env` |
| 非 production | 读 `.env`，再读 `.env.local` |
| 系统环境变量已存在 | `.env` 不覆盖 |
| loader 返回值 | 只返回文件名和 key 名，不返回 value |

`.env` 内容：

```sh
DEEPSEEK_API_KEY=你的本地开发 key
```

注意：`.env` 必须在 `.gitignore` 中。教程、测试、日志都不能出现真实 key。

### AI Prompt：实现 `.env` Loader

```text
请实现 packages/cli/src/devEnv.ts：

1. 导出 loadDevelopmentEnv(options)。
2. NODE_ENV=production 时直接返回，不读取文件。
3. 开发环境依次读取 .env 和 .env.local。
4. 只支持 KEY=VALUE、单引号、双引号、注释即可，不需要完整 dotenv 生态。
5. 系统环境变量优先，.env 不覆盖已存在 key。
6. 返回 mode、files、keys，不要返回任何 value。
7. 添加单测覆盖解析、加载、不覆盖系统变量、production 跳过。
```

## V0.1 到 V0.2 的衔接

V0.2 要开始实现第一个 headless agent loop。它会复用 V0.1 的这些能力：

- 从 CLI 增加 `-p/--print`。
- 用 `model-provider` 发真实 DeepSeek 请求。
- 把 streaming chunk 转成 `QueryEvent`。
- 把 `QueryEvent` 写进 transcript。
- 如果 stop reason 是 `tool_use`，后续版本再进入工具执行。

所以 V0.1 的核心价值不是“功能很多”，而是把后续所有版本都要依赖的协议和 provider 兼容性提前定住。
