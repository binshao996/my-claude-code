# Prompt 设计模式（Prompt Patterns）

Prompt Pattern 的价值不是让文字更漂亮，而是把复杂工程任务变成稳定工作流。

一个 Pattern 应该说明：

- 适用场景。
- 输入契约。
- 输出契约。
- 优势。
- 缺点。
- 失败案例。
- 最佳实践。

## 1. Planner Pattern

### 适用场景

需求不小、影响面不确定、需要先读代码再改。

### 输入契约

- 用户目标。
- 当前仓库或相关目录。
- 已知限制和不做范围。
- 允许读取的文件范围。

### 输出契约

- 相关文件和调用链。
- 当前行为事实。
- 3-6 步计划。
- 风险和验证策略。
- 不修改文件。

### 模板

```text
你现在是 Planner，只读不改。

目标：
{task}

请完成：
1. 找到相关文件和调用链。
2. 列出影响面。
3. 给出 3-6 步实施计划。
4. 标出风险和需要验证的点。

限制：
- 不修改文件。
- 不提出不存在的路径。
- 不做超出目标的重构。

输出：
- Relevant files
- Current behavior
- Plan
- Risks
- Verification
```

### 优势

先把不确定性暴露出来，减少直接实现导致的大返工。

### 缺点

如果 Planner 没读真实代码，会生成空泛计划。

### 失败案例

Planner 直接说“新增 service、controller、repository”，但项目里根本不是这种架构。

### 最佳实践

要求 Planner 引用真实文件路径和当前代码事实。没有证据的计划不能进入实现。

## 2. Architect Pattern

### 适用场景

需要做模块边界、数据流、状态归属、错误策略设计。

### 输入契约

- 目标能力。
- 现有模块和入口。
- 非目标能力。
- API、依赖、权限等硬约束。

### 输出契约

- 模块职责划分。
- 数据流和状态归属。
- 错误、日志、审计策略。
- 分阶段落地方案。
- 明确哪些能力本阶段不实现。

### 模板

```text
你现在是 Architect。

请基于现有代码提出最小架构设计：
- 哪些模块负责输入。
- 哪些模块负责状态。
- 哪些模块负责副作用。
- 错误如何传播。
- 日志/审计如何记录。
- 哪些能力本阶段不实现。

限制：
- 优先复用现有模块。
- 不新增框架。
- 不改变 public API，除非明确说明必要性。
```

### 优势

把概念能力落到真实模块职责上。

### 缺点

容易过度设计。Architect 不应该直接写完整实现。

### 失败案例

为一个简单 token budget 功能设计了 `BudgetDomainService`、`BudgetRepository`、`BudgetController`，但项目只是 CLI 本地工具。

### 最佳实践

要求“最小架构”，并列出“为什么不需要更复杂方案”。

## 3. Executor Pattern

### 适用场景

计划已明确，需要执行小范围代码修改。

### 输入契约

- 已批准计划。
- 允许修改的文件列表。
- 验收命令。
- 遇到计划外问题时的停止规则。

### 输出契约

- 实际修改摘要。
- 文件变更列表。
- 验证命令和结果。
- 阻塞点或残余风险。

### 模板

```text
你现在是 Executor。

按以下计划执行：
{approved_plan}

约束：
- 只修改这些文件：{files}
- 不扩大范围。
- 遇到计划外问题先停止说明。
- 保持现有代码风格。

完成后输出：
- 修改摘要。
- 测试命令和结果。
- 未完成或风险。
```

### 优势

防止实现阶段重新设计。

### 缺点

如果计划本身错了，Executor 会高效执行错误方案。

### 失败案例

计划漏掉了 session restore，Executor 只改 request path，导致 resume 后行为不一致。

### 最佳实践

Executor 遇到真实代码和计划不一致时必须停，不要自己脑补补丁。

## 4. Reviewer Pattern

### 适用场景

实现完成后，需要找 bug、回归、安全风险、测试缺口。

### 输入契约

- 当前 diff。
- 相关文件。
- 预期行为。
- 已运行的测试和结果。

### 输出契约

- Findings first。
- 每个问题包含文件/行号、影响、修复建议。
- 只报告真实风险，不输出泛泛优点。
- 没有问题时说明残余风险和测试缺口。

### 模板

```text
你现在是 Reviewer。

请只读 diff 和相关文件，优先找：
1. 行为 bug。
2. 回归风险。
3. 安全或权限问题。
4. 类型和边界问题。
5. 缺失测试。

不要重写代码。
不要总结优点。
每个问题必须包含：
- 文件/行号。
- 问题。
- 影响。
- 建议修复。
```

### 优势

让模型从“完成任务”切换到“打破实现”。

### 缺点

Reviewer 可能提出风格偏好而非真实问题。

### 失败案例

Review 输出“建议重构命名”十几条，但漏掉权限绕过。

### 最佳实践

明确“Findings first”，只接受可导致 bug、风险或测试缺口的问题。

## 5. Verification Pattern

### 适用场景

非平凡改动交付前，需要证据证明行为正确。

### 输入契约

- 预期行为。
- 改动摘要或 diff。
- 可运行命令范围。
- 环境限制。

### 输出契约

- verdict: PASS / FAIL / PARTIAL。
- 命令证据或无法运行原因。
- 覆盖了什么、没覆盖什么。
- FAIL/PARTIAL 的最小后续动作。

### 模板

```text
你现在是 Verifier，只读验证。

目标：
{expected_behavior}

请：
1. 根据项目选择最小验证命令。
2. 运行或说明无法运行的原因。
3. 检查输出是否真的覆盖目标行为。
4. 给出 verdict: PASS / FAIL / PARTIAL。

PASS 必须包含命令证据。
PARTIAL 必须说明缺口。
FAIL 必须说明最小修复方向。
```

### 优势

避免“看起来完成”直接交付。

### 缺点

验证成本高，可能需要环境准备。

### 失败案例

只运行 typecheck 就给 PASS，但功能链路没有测试。

### 最佳实践

让 Verifier 独立于实现上下文，优先运行和改动最相关的最小测试。

## 6. Refactor Pattern

### 适用场景

代码重复、命名漂移、模块边界变差，但不能破坏行为。

### 输入契约

- 需要保护的行为。
- 当前测试或补测要求。
- 允许修改范围。
- 不允许改变的 API。

### 输出契约

- 重构前的坏味道说明。
- 最小重构步骤。
- 行为不变的证据。
- 回归测试结果。

### 模板

```text
这是 Refactor 任务，不改变行为。

请先列出：
- 当前重复或坏味道。
- 需要保护的行为。
- 最小重构步骤。
- 回归测试。

限制：
- 不新增功能。
- 不改变 public API。
- 每一步都保持测试可通过。
```

### 优势

防止 AI 把重构做成功能重写。

### 缺点

如果没有测试保护，重构风险很高。

### 失败案例

AI “顺手”把错误处理策略改了，导致原本可恢复的工具错误变成进程退出。

### 最佳实践

先加 characterization test，再重构。

## 7. Constraint Pattern

### 适用场景

模型容易自由发挥，需要强约束边界。

### 输入契约

- 允许修改文件。
- 禁止修改文件。
- 禁止新增的能力、依赖或 API 变化。
- 失败时是否停止。

### 输出契约

- 在约束内完成的改动。
- 或明确说明无法完成的原因。
- 不绕过约束，不用伪实现。

### 模板

```text
硬性约束：
- 只能修改 {allowed_files}
- 不新增依赖。
- 不移动文件。
- 不改 public API。
- 不改测试快照，除非行为变化正是目标。
- 生产代码不使用 as any。
- 不输出伪代码。

如果无法在约束内完成，请说明原因并停止。
```

### 优势

显著降低跑偏和过度重构。

### 缺点

约束过死会阻止必要修复。

### 失败案例

禁止改测试但实际类型契约变了，导致只能用错误方式兼容旧测试。

### 最佳实践

约束要配“无法完成时停止说明”，不要逼模型绕规则。

## 8. Incremental Pattern

### 适用场景

大功能需要逐步落地。

### 输入契约

- 完整目标。
- 当前阶段目标。
- 可回滚边界。
- 每个增量的验收方式。

### 输出契约

- 增量拆分。
- 当前只实现的最小增量。
- 后续增量明确延后。
- 当前增量的验证结果。

### 模板

```text
把任务拆成增量，每个增量必须：
- 有独立目标。
- 有最小代码改动。
- 有验证方式。
- 可以回滚。

先实现第一个增量，不做后续增量。
```

### 优势

降低一次失败的损失。

### 缺点

如果增量边界设计不好，后续会返工。

### 失败案例

先实现 UI，再发现底层数据模型不支持，导致 UI 全部重写。

### 最佳实践

增量顺序优先：类型和数据模型 -> 核心行为 -> 集成 -> UI -> 优化。

## 9. Context Isolation Pattern

### 适用场景

探索、实现、Review 相互污染，或者子任务可以独立完成。

### 输入契约

- 子任务目标。
- 必要文件和事实。
- 架构不变量。
- 明确排除的历史噪声。

### 输出契约

- 干净的子任务结果。
- 使用了哪些上下文。
- 不确定点。
- 不把探索过程注入主上下文，只返回结论和证据。

### 模板

```text
为这个子任务创建隔离上下文。

只提供：
- 子任务目标。
- 必要文件。
- 必要约束。
- 输出格式。

不要提供：
- 无关历史讨论。
- 已废弃方案。
- 其他子任务细节。
```

### 优势

减少旧信息干扰，提高子 Agent 稳定性。

### 缺点

隔离过度会丢失关键架构背景。

### 失败案例

子 Agent 不知道项目统一错误类型，自己定义了新错误结构。

### 最佳实践

隔离上下文里保留“架构不变量”，去掉“探索噪声”。

## 10. Spec-driven Pattern

### 适用场景

功能涉及业务规则、边界情况、验收标准。

### 输入契约

- 用户可见目标。
- 业务规则。
- 已知边界。
- 技术约束。

### 输出契约

- Spec 文档。
- 验收测试。
- 非目标场景。
- 开放问题。
- 不写实现代码。

### 模板

```text
先写 Spec，不写代码。

Spec 必须包含：
- 用户可见行为。
- 输入输出。
- 边界情况。
- 错误处理。
- 不支持的场景。
- 验收测试。

Spec 确认后再实现。
```

### 优势

减少“代码写完才发现需求理解错”的风险。

### 缺点

小修小补不需要完整 Spec。

### 失败案例

没有先写权限规则 Spec，AI 把 deny 当成异常抛出，结果 Agent Loop 直接中断。

### 最佳实践

Spec 不是产品文档，要能直接推导测试。

## 11. Workflow-driven Pattern

### 适用场景

任务跨多个阶段，单轮 Prompt 不稳定。

### 输入契约

- 任务目标。
- 风险等级。
- 允许执行的动作。
- 每个阶段的停止条件。

### 输出契约

- 每个阶段的结果。
- 阶段间发现的前提变化。
- 最终修改、验证和风险。
- 前提错误时停止，不继续硬做。

### 模板

```text
按固定工作流执行：
1. Inspect：读取相关代码。
2. Plan：列计划。
3. Patch：小范围修改。
4. Test：运行最小检查。
5. Review：检查 diff。
6. Report：说明结果和风险。

每个阶段结束后，如果发现前提错误，停止并报告。
```

### 优势

把 Agent 行为从“生成答案”变成“执行流程”。

### 缺点

流程过重会拖慢简单任务。

### 失败案例

简单文案修改也套完整流程，浪费时间。

### 最佳实践

按风险选择流程重量。越接近权限、数据、构建、发布，流程越严格。

## 12. Debug Hypothesis Pattern

### 适用场景

报错原因不明确，需要实验排查。

### 输入契约

- 具体现象。
- 错误日志或复现步骤。
- 最近改动。
- 可运行的最小实验范围。

### 输出契约

- Hypothesis。
- Experiment。
- Result。
- Fix。
- 没有证据时不提交修复。

### 模板

```text
请按调试假设工作：
1. 列出 3 个最可能原因。
2. 每个原因对应一个最小实验。
3. 先运行成本最低的实验。
4. 根据结果收敛假设。
5. 只修复被证据支持的问题。

输出必须包含：
- Hypothesis
- Experiment
- Result
- Fix
```

### 优势

避免 AI 猜修。

### 缺点

需要可运行环境和足够日志。

### 失败案例

AI 看到 TypeScript 报错就改类型断言，但真实原因是 import 路径错。

### 最佳实践

先复现，再修复。不能复现时说明证据缺口。

## 13. Output Contract Pattern

### 适用场景

需要稳定解析、稳定 Review、稳定交付。

### 输入契约

- 目标读者：人类、脚本、Reviewer 或发布记录。
- 必须出现的字段。
- 是否允许额外说明。
- 失败时输出格式。

### 输出契约

- 严格按指定结构输出。
- 不漏字段。
- 不混入无关章节。
- 无法满足时输出明确错误状态。

### 模板

```text
输出格式固定：

Summary:
- ...

Files changed:
- path: reason

Verification:
- command: result

Risks:
- ...

No extra sections.
```

### 优势

减少废话，便于后续自动化和人工审查。

### 缺点

过度固定会压制必要解释。

### 失败案例

要求 JSON 输出，但任务需要解释权衡，模型为了 JSON 合法省略风险。

### 最佳实践

机器消费用严格格式；人类决策用结构化 Markdown。

## 14. Prompt Refresh Pattern

### 适用场景

长会话后期模型开始遗忘、混淆、重复旧错误。

### 输入契约

- 当前目标。
- 最近有效 diff 或文件状态。
- 已完成和未完成事项。
- 废弃方案。
- 最新用户约束。

### 输出契约

- 可作为新上下文的状态摘要。
- 保留关键工程事实。
- 删除探索噪声。
- 明确下一步动作。

### 模板

```text
请基于当前事实重建任务上下文：
- 当前目标。
- 已完成。
- 未完成。
- 关键文件。
- 架构决策。
- 废弃方案。
- 最新用户约束。
- 下一步。

压缩后，后续只以这份摘要为准。
```

### 优势

降低历史噪声，恢复输出质量。

### 缺点

摘要可能丢信息。

### 失败案例

摘要只写“权限系统已完成”，没写权限规则格式，后续实现无法接上。

### 最佳实践

摘要必须保留工程状态，不只保留自然语言结论。

## 15. Stop-and-Ask Pattern

### 适用场景

任务涉及高风险决策，AI 不应擅自决定。

### 输入契约

- 高风险决策列表。
- 允许 AI 自主处理的低风险范围。
- 需要用户确认时的输出格式。

### 输出契约

- 触发条件时停止。
- 给出 2-3 个可选方案。
- 说明每个方案的成本和风险。
- 不擅自执行不可回滚操作。

### 模板

```text
以下情况必须停止并询问：
- 新增依赖。
- 改 public API。
- 删除数据或迁移格式。
- 修改认证、权限、密钥、网络访问。
- 大范围重构。
- 无法找到用户指定文件。
```

### 优势

保护架构和安全边界。

### 缺点

如果触发条件太宽，会频繁打断。

### 失败案例

AI 为了实现一个小功能新增数据库依赖，团队并不需要。

### 最佳实践

把 Stop-and-Ask 用在不可轻易回滚的决策上，小的实现细节交给 AI 自主处理。
