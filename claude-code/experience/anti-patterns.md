# AI 工程协作反模式（Anti-patterns）

反模式不是“写得不够优雅”，而是会系统性降低 AI 工程成功率的协作方式。

每个反模式都包含：

- 为什么危险。
- 如何识别。
- 如何避免。
- 如何修复。

## 1. 超长 Prompt 反模式

### 表现

把所有背景、规则、历史、示例、偏好、错误日志都塞进一个 Prompt。

### 为什么危险

长 Prompt 会带来注意力稀释和指令冲突。越往后追加，越容易出现旧约束没删除、新约束不生效、临时调试规则污染长期规则。

### 如何识别

- Prompt 超过几千 token，但任务本身很小。
- 同一约束重复出现多次。
- 里面有已经过期的方案。
- 模型开始忽略最新要求。

### 如何避免

分层维护 Prompt：全局规则、项目规则、任务规则、输出契约分开。

### 如何修复

做 Prompt Refresh：只保留当前目标、有效约束、相关文件、验收标准。删除探索过程和过期方案。

## 2. 不做任务拆分

### 表现

让 AI 一次性完成系统设计、实现、测试、文档、发布。

### 为什么危险

任何早期误解都会被后续所有输出放大。任务越大，模型越倾向生成概述、伪代码或玩具架构。

### 如何识别

- 一个 Prompt 里有多个动词：设计、实现、重构、测试、部署。
- 输出文件超过 10 个且没有阶段计划。
- AI 开始“顺手”改无关模块。

### 如何避免

按 Spec -> Plan -> Code -> Test -> Review 拆阶段。

### 如何修复

停止当前大任务，回到一个最小闭环。先让 AI 列出已改文件和风险，再决定保留哪些改动。

## 3. 不做上下文隔离

### 表现

探索、实现、调试、Review 都在同一个长会话里进行。

### 为什么危险

探索阶段的错误假设会污染实现阶段；实现阶段的自我辩护会污染 Review 阶段。

### 如何识别

- AI 反复引用已经否定的方案。
- Review 只是在解释实现为什么合理。
- 子任务输出包含大量无关历史。

### 如何避免

Planner、Executor、Reviewer 使用不同上下文。子 Agent 只拿必要文件和约束。

### 如何修复

生成一份当前事实摘要，另开隔离上下文继续。摘要里明确“废弃方案”。

## 4. Prompt 不可维护

### 表现

Prompt 像补丁一样不断追加：“再注意一下”“以后不要”“顺便还要”。

### 为什么危险

长期看会变成冲突规则集合。模型不知道哪些是稳定规则，哪些是临时要求。

### 如何识别

- Prompt 里有大量“另外”“补充”“注意”。
- 同一项目不同任务的规则互相矛盾。
- 修改 Prompt 后没有删除旧规则。

### 如何避免

把 Prompt 当工程文档维护，模块化、版本化、定期清理。

### 如何修复

重写 Prompt 架构：Role、Constraints、Workflow、Context、Output Contract 分段。

## 5. 过度相信 AI

### 表现

AI 说完成就认为完成，AI 说测试应该通过就不运行测试。

### 为什么危险

模型会给出高置信但错误的判断。它擅长解释，不等于擅长验证。

### 如何识别

- 最终回复没有命令证据。
- 没有 diff Review。
- 没有失败路径测试。
- AI 使用“应该”“看起来”描述完成状态。

### 如何避免

非平凡改动必须有验证命令、测试输出或人工验收证据。

### 如何修复

引入 Verification Pattern。没有 PASS 证据的交付只能标记 PARTIAL。

## 6. 没有约束输出

### 表现

让 AI 自由回答，最后得到长篇解释、伪代码、无结构总结。

### 为什么危险

不可解析、不可审查、不可追踪。下一轮很难基于它继续工作。

### 如何识别

- 输出没有文件列表。
- 输出没有测试结果。
- 输出没有风险。
- 输出包含大量泛泛建议。

### 如何避免

使用 Output Contract，固定 Summary、Files changed、Verification、Risks。

### 如何修复

让 AI 重新按契约整理已有结果，不要继续扩写。

## 7. 没有工程规范

### 表现

只要求“能跑”，不要求类型、错误处理、日志、测试、目录结构。

### 为什么危险

AI 会生成 Demo Code。短期能展示，长期难维护。

### 如何识别

- 生产代码出现 `as any`。
- 错误被吞掉。
- 大输出直接塞进上下文。
- 新模块没有被主链路调用。
- 没有测试。

### 如何避免

把工程质量写成硬性约束：类型、错误、日志、审计、测试、预算。

### 如何修复

先补测试和调用链，再清理类型和错误处理。不要先做美化重构。

## 8. 一次性生成整个系统

### 表现

一个 Prompt 生成 CLI、Agent、Tools、Memory、Plugin、Remote Control。

### 为什么危险

模型会生成互相不连接的概念模块。文件很多，但没有真实闭环。

### 如何识别

- 目录看起来完整，但启动路径没接上。
- 类很多，调用很少。
- 测试只测纯函数，不测主流程。

### 如何避免

从最小闭环开始：输入 -> 模型 -> 工具 -> 结果 -> 第二轮模型 -> 输出。

### 如何修复

找主链路，删除或暂存未接入模块，优先打通一个端到端路径。

## 9. 不做阶段化开发

### 表现

AI 在同一轮里读代码、设计、实现、修错、写总结。

### 为什么危险

没有检查点，无法及时纠偏。

### 如何识别

- 计划和实现混在一起。
- 没有中间验收。
- 失败后直接大改。

### 如何避免

每个阶段有明确进入条件和退出条件。

### 如何修复

暂停实现，要求 AI 输出当前状态、偏离点、下一步最小动作。

## 10. 用抽象词代替真实约束

### 表现

“写得优雅”“架构清晰”“工程级”“不要跑偏”。

### 为什么危险

模型会用表面形式满足抽象词，比如加更多类、更多注释、更多层。

### 如何识别

- 输出看起来更复杂，但行为没有更可靠。
- 新增抽象没有减少真实重复。
- 没有对应测试。

### 如何避免

把抽象词翻译成可检查规则：文件范围、API 不变、错误策略、测试命令。

### 如何修复

让 AI 列出每个新增抽象解决的具体重复或复杂度；说不清就删除。

## 11. 让 AI 主导架构决策

### 表现

“你觉得怎么设计就怎么设计。”

### 为什么危险

AI 不承担维护成本，也不知道组织约束。它可能选择看起来先进但团队无法维护的方案。

### 如何识别

- 新增依赖没有理由。
- 改 public API 没有迁移方案。
- 安全、权限、数据格式被顺手改动。

### 如何避免

高风险决策必须 Stop-and-Ask。

### 如何修复

回到人类决策点：列方案、成本、风险、迁移路径，由人类批准。

## 12. 忽略真实调用链

### 表现

AI 写了新模块，但没有接入入口。

### 为什么危险

代码存在不代表功能存在。未接入模块会制造虚假的完成感。

### 如何识别

- 新文件没有 import 使用。
- 测试只覆盖新函数，不覆盖业务入口。
- 用户路径仍然走旧逻辑。

### 如何避免

要求每个功能说明调用链：入口、状态、执行、副作用、输出。

### 如何修复

先写集成测试暴露未接入问题，再接主链路。

## 13. 把日志当上下文

### 表现

把完整 debug log、terminal output、transcript 全部塞给模型。

### 为什么危险

日志里有噪声、重复、过期错误，甚至可能有敏感信息。模型会被无关输出带偏。

### 如何识别

- 上下文里充满 stack trace 但没有当前问题摘要。
- 模型修复已经不存在的错误。
- Token 快速爆炸。

### 如何避免

先提取最小错误片段：命令、错误、相关文件、复现步骤。

### 如何修复

让 AI 根据日志生成“事实摘要”，下一轮只使用摘要和必要片段。

## 14. 把 Memory 当垃圾桶

### 表现

把所有用户偏好、临时任务、调试结论、失败尝试都写进长期记忆。

### 为什么危险

Memory 会持续污染后续任务。临时规则一旦长期化，后续很难察觉。

### 如何识别

- AI 在无关任务里引用旧任务规则。
- Memory 里有具体 bug 的过期信息。
- 用户偏好和项目事实混在一起。

### 如何避免

Memory 分层：用户偏好、项目规则、长期架构决策、当前任务状态分开。

### 如何修复

定期清理 Memory。删除临时调试信息，只保留稳定决策。

## 15. 用自检替代 Review

### 表现

实现后让同一个 AI “检查一下”，然后直接交付。

### 为什么危险

同一上下文会继承实现时的假设。模型容易解释自己的代码，而不是攻击它。

### 如何识别

- 自检没有具体问题。
- 全是“看起来正确”。
- 没有运行命令。

### 如何避免

Review Prompt 必须独立，只读 diff，优先找 bug。

### 如何修复

重新发起 Reviewer Pattern，要求 Findings first，并给出文件/行号。

## 16. Token 预算失控

### 表现

大文件、长日志、完整工具输出、完整历史无限进入上下文。

### 为什么危险

成本升高，响应变慢，注意力稀释，甚至 prompt-too-long。

### 如何识别

- 每轮输入越来越慢。
- 模型开始漏掉当前目标。
- 工具结果占据主要上下文。

### 如何避免

对 memory、runtime context、messages、tool result 分别设预算。大输出截断或写 side file。

### 如何修复

MicroCompact 旧工具结果，Compact 旧会话，保留最近关键消息和决策摘要。

## 17. 权限和安全后补

### 表现

先让 Agent 能执行所有工具，后面再加权限。

### 为什么危险

权限是调用链核心，不是 UI 装饰。后补权限容易漏掉后台任务、子 Agent、远程执行和插件入口。

### 如何识别

- 工具 execute 里没有权限上下文。
- 后台 Agent 可以绕过确认。
- 审计日志没有 deny/ask 记录。

### 如何避免

从 Tool Context 设计阶段就带上 permission mode、scope、audit。

### 如何修复

统一工具执行入口，在入口处做权限判断和审计，不要在每个工具里零散处理。

## 18. 只追求生成速度

### 表现

不断要求“快点继续”，不做测试、不做复盘、不清理上下文。

### 为什么危险

短期速度换来长期返工。AI 会复制最近错误，质量逐轮退化。

### 如何识别

- 同类 bug 反复出现。
- 命名越来越不一致。
- TODO 和伪代码增多。
- Review 问题越来越基础。

### 如何避免

每完成一个阶段做架构回收和 Prompt Refresh。

### 如何修复

暂停新增功能，先做质量回收：测试、类型、重复代码、未接入模块、过期规则。

## 总结

AI 工程协作失控通常不是某一次回答错了，而是流程长期缺少边界：

- 没有任务边界。
- 没有上下文边界。
- 没有权限边界。
- 没有架构边界。
- 没有验证边界。

要修复这些问题，不要继续追加“请认真一点”。应该把协作方式工程化：拆任务、隔离上下文、固定输出契约、接入真实调用链、用验证门禁收尾。
