# AI Coding Agent 工程经验与 Prompt 调教实战

这份文档总结的是长期用 AI 参与复杂工程系统开发之后，真正会影响成功率的经验。

它不是“怎么把问题问清楚”的入门文章。复杂工程里的 AI 协作，关键不在某一句神奇 Prompt，而在四件事：

1. 把任务变成可执行、可验证、可回滚的工程单元。
2. 把上下文变成经过选择、压缩、隔离、刷新后的高质量输入。
3. 把模型输出放进 Planner、Executor、Reviewer、Verifier 这样的工作流。
4. 人类始终控制架构边界、质量门禁和最终决策。

Claude Code 类系统的核心经验是：AI Coding Agent 不是“聊天框里更聪明的补全”，而是一条由 Prompt、Context、Tool Calling、权限、日志、压缩、记忆、验证和人工 Review 共同约束的工程流水线。

## 1. 如何写高质量 Prompt

高质量 Prompt 不是长，而是结构稳定、职责清晰、可维护。

复杂工程 Prompt 至少要分层：

- Role Prompt：定义角色和能力边界。
- Constraint Prompt：定义禁止项、风格、权限、范围。
- Workflow Prompt：定义先做什么、后做什么、何时停。
- Context Prompt：提供当前项目事实、文件、接口、历史决策。
- Output Contract：定义输出格式、验收证据和失败时怎么说。
- Planning Prompt：让模型先拆解任务、识别风险。
- Reflection Prompt：让模型在提交前自检，但不能替代真实验证。

### 错误示例

```text
帮我实现一个类似 Claude Code 的 Agent，要支持工具调用、上下文、记忆、插件、权限、远程控制，代码要优雅。
```

### 正确示例

```text
目标：在现有 Mini Agent 中实现 Tool Calling 的最小闭环。

范围：
- 只实现 read_file 和 write_file 两个工具。
- 不实现权限系统，不实现 MCP，不实现多 Agent。
- 必须复用现有 ChatSession 和 ModelClient。

工作流：
1. 先阅读 src/agent、src/model、src/tools 的现有结构。
2. 给出 3-5 步实现计划。
3. 修改最少文件。
4. 增加一个单元测试覆盖 tool_use -> tool_result -> second model call。
5. 最后说明运行过的命令和结果。

输出契约：
- 如果发现现有模块不存在，先说明，不要发明路径。
- 不允许输出伪代码。
- 不允许重构无关模块。
```

### 为什么错误

错误 Prompt 把目标、范围、架构、质量要求混在一句话里。模型会自动补全缺失部分，倾向生成“大而全”的玩具架构，甚至编造不存在的模块。

### 为什么正确

正确 Prompt 把任务限定在一个可验证闭环里，明确了不做什么、复用什么、怎么验收、失败时如何处理。它降低了模型自由发挥空间。

### 最佳实践

- 先写“范围”和“不做什么”，再写实现要求。
- 任何大型任务都要求 AI 先读代码再计划。
- Output Contract 要具体到文件、命令、测试、失败格式。
- 长期 Prompt 要模块化，不要把所有规则追加到一个巨型段落。

### 实战经验

Prompt 最容易坏在后期追加规则。今天加“不要重构”，明天加“要详细解释”，后天加“输出完整代码”，最后规则互相冲突。稳定做法是分层维护：全局工程规则、项目规则、当前任务规则、输出契约分开写。

### 真实踩坑

在实现 Context Compaction 时，如果只说“帮我总结历史上下文”，模型经常会丢掉工具调用配对、文件路径和未完成决策。后来把 Prompt 改成“必须保留当前目标、已修改文件、测试结果、失败原因、用户明确约束、下一步动作；禁止保留无关闲聊”，压缩后的会话恢复质量才稳定。

## 2. 如何避免 AI 跑偏

AI 跑偏通常不是因为“它不听话”，而是输入里缺少可执行边界。

模型会根据概率补全最像答案的内容。当上下文里没有真实文件、真实 API、真实测试、真实验收标准时，它会用训练中常见的工程套路填空。

### 错误示例

```text
优化一下这个项目的架构。
```

### 正确示例

```text
只修复 src/context/compact.ts 中 compact 后 tool_use/tool_result 配对可能被切断的问题。

限制：
- 不改 public API。
- 不移动文件。
- 不重写 compact 策略。
- 只允许新增必要的 helper 和测试。

验收：
- 新增测试覆盖 assistant tool_use 被保留但 user tool_result 被裁掉的情况。
- bun run typecheck 通过。
```

### 为什么错误

“优化架构”没有边界。模型会把局部问题上升成系统重构，生成大量看起来合理但没有必要的抽象。

### 为什么正确

正确 Prompt 把跑偏空间压缩到一个文件、一个 bug、一个测试场景。模型没有理由去改插件、权限、UI 或模型路由。

### 最佳实践

- 每个任务都写“禁止项”：禁止新增依赖、禁止改 public API、禁止移动目录。
- 要求 AI 引用真实文件路径，不能“假设存在”。
- 对复杂任务先让 AI 输出候选影响面，再决定是否执行。
- 大改必须分阶段：定位、计划、实现、验证、复盘。

### 实战经验

最有效的约束不是“不要跑偏”，而是“只允许改哪些文件”和“用什么证据证明完成”。AI 对抽象禁止语的遵守不稳定，但对具体文件边界、测试命令、输出格式更稳定。

### 真实踩坑

有一次让 AI “清理工具系统”，它把 Tool Registry、权限判断、工具 schema、日志输出一起重构，最后 typecheck 爆了几十个错。修复方式是回退大改，重新拆成三个任务：先修类型，再统一 schema，再补权限日志。每个任务只允许改 2-4 个文件。

## 3. 如何让 AI 更懂工程

AI 默认擅长生成“看起来像工程”的代码，不等于能生成“能长期维护的工程代码”。

让 AI 更懂工程，核心是要求它基于真实调用链工作：

- 谁调用这个模块。
- 输入输出类型是什么。
- 错误如何传播。
- 状态在哪里持久化。
- 日志和审计写在哪里。
- 测试如何证明行为。

### 错误示例

```text
写一个记忆系统，支持长期记忆和短期记忆。
```

### 正确示例

```text
在现有 Agent Loop 中增加 Memory 注入，不实现复杂向量检索。

真实调用链：
- CLI 启动时加载 project memory 和 user memory。
- ContextPreparer 在每次模型请求前注入 memory。
- TokenBudget 对 memory 设置独立上限。
- Transcript 只记录 memory 版本信息，不记录隐私原文。

必须说明：
- memory 文件路径从哪里读。
- 超预算时如何裁剪。
- 测试覆盖哪些行为。
```

### 为什么错误

“长期记忆/短期记忆”是概念，不是工程接口。模型容易输出 `MemoryManager`、`VectorStore`、`EmbeddingService` 这样的概念代码，和项目主链路脱节。

### 为什么正确

正确 Prompt 把记忆系统接入启动、上下文准备、预算、transcript 四个真实模块。模型必须考虑边界和副作用。

### 最佳实践

- 要求 AI 画出或列出调用链，再写代码。
- 要求区分“本章实现”和“后续保留接口”。
- 要求错误处理策略：抛错、降级、跳过、记录。
- 对状态类功能必须明确持久化格式和恢复流程。

### 实战经验

工程级代码不是“多封装几个类”。真正的工程性来自边界稳定：模块职责单一、输入输出明确、失败可观测、行为可测试。AI 如果没有被要求处理这些点，会自然退回 Demo Code。

### 真实踩坑

在做 Session Resume 时，AI 曾经把 debug log 当成 transcript 恢复来源。看起来能恢复文本，但一旦里面有错误日志、权限事件、半截 streaming token，会污染 message chain。最后必须拆成三套：debug log、runtime event、transcript JSONL。

## 4. 如何与 AI 协作开发大型项目

大型项目比小 Demo 难，不是代码更多，而是历史决策更多、模块边界更多、上下文更容易漂移。

AI 在小任务里只需要“生成正确代码”。在大型工程里，它必须持续遵守过去几十轮的架构约束，这对模型很难。

### 错误示例

```text
继续完善这个项目，把剩余能力都补齐。
```

### 正确示例

```text
本阶段只推进第 34 章：验证 Agent 与交付门禁。

阶段目标：
1. 定义 verification 类型和 verdict。
2. 实现只读 verifier prompt。
3. 增加 delivery gate。
4. 写 verdict 解析和 gate 测试。

保持一致：
- Agent 定义格式沿用第 30-31 章。
- 持久化沿用 transcript/store 的 JSONL 风格。
- 命令风格沿用现有 slash command。
```

### 为什么错误

“剩余能力”会让模型自选优先级。它可能同时碰插件、远控、权限、UI，导致上下文和改动面失控。

### 为什么正确

正确 Prompt 把大型项目拆到一个章节级能力，并把它连接到已有章节的架构约束。

### 最佳实践

- 用阶段文档记录术语、架构决策和不变量。
- 每轮只推进一个可交付闭环。
- 长项目定期做 Context Refresh：把当前事实重写成短摘要。
- 保留“已决定不做”的清单，防止 AI 反复引入旧方案。

### 实战经验

长期协作中最重要的不是“记住所有细节”，而是维护一份高质量状态摘要：当前目标、已完成、未完成、关键约束、风险、下一步。这个摘要比完整聊天历史更可靠。

### 真实踩坑

多轮写教程时，AI 后期会把“Mini 当前不实现”的能力写成“本章已经实现”。修复方式是在每章 Prompt 里固定加入“本章不做什么”和“与官方差距”，并要求最后验收标准只写本章真实完成的能力。

## 5. 如何调教 Claude / GPT / Gemini

不要把模型选择变成品牌信仰。模型能力会随版本变化，工程上应该按任务类型、上下文长度、工具稳定性、输出风格和验证成本选择。

通用经验：

- Claude 更适合长文档理解、复杂代码库阅读、工程约束遵守、重构计划、长上下文总结。
- GPT 更适合结构化推理、工具编排、代码解释、测试补全、快速生成多种方案。
- Gemini 在大上下文、多模态输入、文档横向扫描、快速摘要类任务上经常有优势。

具体项目里必须用实际任务基准验证，不要永久绑定某个模型。

### 错误示例

```text
这个任务很复杂，用最强模型一次性写完。
```

### 正确示例

```text
模型分工：
- 用长上下文模型阅读 20 个相关文件，产出影响面摘要。
- 用代码能力强的模型实现局部改动。
- 用独立 Reviewer/Verifier 模型只读检查 diff、运行测试、找回归。
```

### 为什么错误

复杂任务不是模型越强越能一次完成。一次性输入过多，会引入上下文污染、注意力稀释和输出失焦。

### 为什么正确

正确方式把模型当成流水线里的不同角色，而不是单次万能调用。

### 最佳实践

- 长文档分析和上下文压缩优先选长上下文稳定的模型。
- 小范围代码修改优先选遵守指令强、工具调用稳定的模型。
- UI 生成可以用创意强的模型，但必须人工视觉验收。
- Debug 任务要求模型输出假设、实验、结果，不要只给修复。
- Reviewer 最好独立于 Implementer，避免同一上下文自证正确。

### 实战经验

模型组合最有效的场景是“读”和“改”分离。一个模型负责读仓库和产出事实摘要，另一个模型只基于摘要和目标改代码，第三个模型做只读 Review。这样比把所有历史塞给一个模型稳定。

### 真实踩坑

曾经用一个长上下文会话连续完成设计、实现、修 bug、写文档。到后期模型开始复用早期错误命名，并忽略新的目录结构。后来改成每个阶段重建上下文，只保留决策摘要和当前 diff，质量明显恢复。

## 6. 如何让 AI 输出高质量代码

AI 默认输出 Demo Code，是因为训练语料里大量示例代码追求“展示概念”，不是“长期维护”。

纠正方式是把工程质量写进验收条件，而不是泛泛说“代码要优雅”。

### 错误示例

```text
代码写得专业一点，注意可维护性。
```

### 正确示例

```text
质量要求：
- 类型不能使用生产代码 as any。
- 新增错误分支必须返回结构化错误或抛出带上下文的 Error。
- 工具执行结果必须写入 transcript。
- 大输出必须截断或写 side file。
- 新行为必须有单元测试。
- 不允许新增全局 mutable state，除非说明生命周期。
```

### 为什么错误

“专业”“可维护”是抽象偏好，模型会用表面形式回应，比如加类、加注释、加接口。

### 为什么正确

正确要求把质量落到类型、错误、日志、输出大小、状态生命周期、测试这些可检查项。

### 最佳实践

- 明确类型策略：unknown、type guard、discriminated union。
- 明确错误策略：用户错误、系统错误、工具错误分开。
- 明确日志策略：debug log 不进入 transcript，审计事件要脱敏。
- 明确测试策略：纯函数单测、集成链路测试、回归测试。

### 实战经验

让 AI 写高质量代码，最有效的是给它现有代码风格和反例。比如告诉它“这个项目中 Tool 是 registry + schema + execute，不要新增独立 ToolManager”，比说“保持架构一致”更有用。

### 真实踩坑

AI 很喜欢在测试里 mock 掉真正应该测试的纯函数，导致测试通过但行为没被验证。后来测试规范改成“只 mock 有副作用的依赖链，不 mock 纯函数/纯数据模块”，测试质量才提升。

## 7. 如何让 AI 写长教程

AI 写长教程容易崩，原因是它很难跨几十章保持同一套术语、代码风格、能力边界和章节节奏。

长教程不是“连续生成文章”，而是“维护一套教学协议”。

### 错误示例

```text
继续写下一章，保持风格一致。
```

### 正确示例

```text
继续写第 45 章。

固定结构：
- 本章目标
- 完成效果
- 与前一章关系
- 真实工程怎么做
- 推荐目录
- 分步骤实现
- 常见坑
- 与官方差距
- 验收标准
- 小结

一致性要求：
- Mini 当前实现和官方能力必须分开。
- 所有代码路径必须是本章推荐目录里的真实路径。
- 不允许提前实现后续章节能力。
```

### 为什么错误

“保持风格一致”没有给模型可复用结构。长上下文后期，模型会只记得大概语气，忘记章节契约。

### 为什么正确

正确 Prompt 固定章节骨架、能力边界和术语规则。模型只需要填充当前章节内容。

### 最佳实践

- 为教程维护全局术语表。
- 每章固定“本章不做什么”。
- 代码风格和目录结构持续复用。
- Diagram 风格固定：同类流程用同类图。
- 每 5-10 章做一次目录和能力清单复盘。

### 实战经验

写长教程时，不要让 AI 从完整历史里找一致性。应该给它一个短的“教程写作规范”和当前章节上下文。完整历史越长，越容易把废弃方案带回来。

### 真实踩坑

早期章节说“暂不实现权限”，后面 AI 回看上下文时误以为权限系统已经存在，于是代码直接调用 `PermissionStore`。解决方式是在每章开头写“当前已有能力”和“本章新增能力”，防止模型跨阶段脑补。

## 8. 如何管理长上下文

Context 是 AI 工程的核心瓶颈。

不是 Prompt 越长越好。长上下文会带来注意力稀释、历史噪声、冲突指令、过期事实、工具结果膨胀和摘要失真。

### 错误示例

```text
把所有历史聊天、所有源码、所有日志都发给模型，让它自己判断。
```

### 正确示例

```text
上下文分层：
- System：稳定角色、工具规则、安全边界。
- Project：架构不变量、目录、测试命令、术语表。
- Task：本轮目标、允许修改范围、验收标准。
- Evidence：相关文件片段、错误日志、测试输出。
- Memory：长期决策摘要，不放临时探索过程。

预算：
- 大工具结果只给摘要和引用。
- 旧历史压缩成任务状态。
- 当前 diff 和失败测试优先级最高。
```

### 为什么错误

全量输入会污染注意力。模型可能抓住早期废弃方案、旧错误日志或无关工具输出，反而忽略当前目标。

### 为什么正确

正确做法按用途组织上下文，并给每层预算。模型更容易识别哪些信息稳定、哪些信息当前有效。

### 最佳实践

- Context Packing：把相关事实压到一起，减少跳跃。
- Context Isolation：探索、实现、Review 分开上下文。
- Context Refresh：阶段结束后重写当前状态。
- Context Compression：压缩旧历史时保留工程决策和未完成事项。
- Memory Layering：用户偏好、项目规则、任务状态分层存储。
- Retrieval Injection：只注入与任务相关的片段。
- Context Window Budgeting：给 system、memory、runtime、messages、tool result 分别设预算。

### 实战经验

最有价值的上下文不是“更多”，而是“更干净”。一次失败的工具调用，如果后来已经修复，就不要让它长期留在主上下文里。保留最终结论和证据引用即可。

### 真实踩坑

实现 auto compact 时，如果直接按消息数量裁剪，会切断 `tool_use` 和 `tool_result` 配对，导致恢复后模型看到一个没有结果的工具调用。后来改成按 API round 分组，并在 compact boundary 里记录 preserved segment，才稳定。

## 9. 如何调试 AI Agent

调试 AI Agent 不能只看最终回答，要看 Agent Loop 的每个状态转移。

你需要能回答：

- 模型收到了什么 system/context/messages/tools。
- 模型为什么选择这个 tool。
- tool input 是否符合 schema。
- tool result 是否被正确写回。
- 权限是否改变了执行路径。
- retry/fallback 是否发生。
- compact/memory 是否改变了上下文。

### 错误示例

```text
Agent 没按预期工作，帮我修一下。
```

### 正确示例

```text
现象：Agent 调用了 write_file，但第二轮模型没有看到 tool_result。

请按顺序检查：
1. streaming 事件里是否收到了 tool_use。
2. Tool Registry 是否找到 write_file。
3. execute 是否返回结构化 tool_result。
4. AgentLoop 是否把结果 append 到 messages。
5. 下一次 model call 的 messages 是否包含 tool_result。

输出：给出最可能断点、证据、最小修复。
```

### 为什么错误

“没按预期”缺少链路位置。AI 可能从 Prompt、模型、工具、UI 随机猜。

### 为什么正确

正确 Prompt 把 Agent Loop 拆成可观察节点，让调试从猜测变成排查。

### 最佳实践

- Agent Loop 必须有 transcript。
- Tool Calling 必须记录 tool name、input 摘要、结果状态、耗时。
- 权限系统必须记录 allow/deny/ask 的原因。
- Context Preparer 必须输出 token 分类和裁剪信息。
- Debug Prompt 时要保留“模型实际收到的输入”，而不是人类以为发了什么。

### 实战经验

调试 Agent 最怕“黑盒成功”。一次调用成功不代表链路正确。必须故意制造失败：工具报错、权限拒绝、输出超长、API fallback、compact 后恢复，看系统是否能继续。

### 真实踩坑

一个工具调用 bug 表现为“模型反复调用同一个工具”。最初以为 Prompt 不够强，后来查 transcript 发现 tool_result 的 `tool_use_id` 没对上，模型其实没收到上一轮结果。修复 ID 关联后，Prompt 不用改。

## 10. 真实踩坑总结

下面这些坑在复杂 AI 工程里非常常见。

### 错误示例

```text
为了省时间，让 AI 一次性生成系统、测试、文档和部署脚本。
```

### 正确示例

```text
分四轮：
1. Spec：定义最小闭环和不做范围。
2. Plan：读代码，列影响面。
3. Code：实现一个闭环，补测试。
4. Review：独立检查 diff、运行命令、修复。
```

### 为什么错误

一次性生成会把需求理解、架构设计、实现、验证混在同一上下文里。任何早期误解都会被后续代码放大。

### 为什么正确

阶段化工作流允许在每个阶段纠偏，减少返工。

### 常见坑

- 错误 Prompt：目标太大、范围太宽、验收太虚。
- AI 跑偏：没有文件边界和禁止项。
- 错误架构：概念类太多，真实调用链没接上。
- Context 崩坏：旧日志、废弃方案、失败探索长期留在上下文。
- Tool Calling 崩坏：schema 和执行结果不一致。
- Token 爆炸：大文件、大日志、全量 transcript 直接塞模型。
- Prompt 污染：临时调试规则进入长期规则。
- 多轮协作失控：没有阶段摘要和决策记录。

### 最佳实践

- 每轮只解决一个工程问题。
- 每个结论都要有代码路径、测试或日志证据。
- 定期清理上下文，不把探索过程当长期记忆。
- 大任务必须有 Reviewer 或 Verification Gate。

### 实战经验

真实工程里，AI 最危险的输出不是明显错误，而是“看起来很合理但没接上系统”。比如新增一个漂亮的 `PolicyEngine`，但没有接入权限判断入口；新增一个 `MemoryStore`，但 ContextPreparer 从未读取它。

### 真实踩坑

有次实现插件供应链校验时，AI 写了 signature verifier，但安装流程仍然直接复制插件目录。测试只覆盖 verifier 函数，没有覆盖 install path。最后通过端到端测试才发现校验完全没有进入主链路。

## 11. AI 的行为机制

理解 AI 为什么会犯错，比背 Prompt 技巧更重要。

大模型本质上是在上下文和训练分布上做 next-token prediction。它会倾向输出“在语料里常见、在当前上下文里顺滑”的内容，而不是自动追求真实工程正确性。

### 错误示例

```text
AI 忽略约束，是因为它不认真。多强调几遍就行。
```

### 正确示例

```text
AI 忽略约束，通常是因为：
- 约束在长上下文后部，注意力权重变弱。
- 当前任务和旧规则冲突。
- 示例代码暗示了另一种行为。
- 输出格式没有可验证契约。
- RLHF 倾向让模型给出“有帮助”的答案，而不是停下来拒绝脑补。
```

### 为什么错误

把问题归因成“态度”会导致继续追加警告，Prompt 越来越长，污染越来越重。

### 为什么正确

正确归因会推动你改变上下文结构、指令层级和验证机制。

### 行为解释

- 忽略约束：长上下文下注意力分散，后置约束更容易被弱化。
- 脑补：缺少事实时，模型用高概率工程套路补全。
- 偷懒：当任务过大或输出契约模糊时，模型会给概述或伪代码。
- 风格漂移：历史样式、当前示例和训练分布混合，长期输出逐渐偏移。
- Demo Code：训练语料中教程和示例代码占比高，默认更像演示。
- 大重构：抽象任务没有边界，模型会追求全局一致性的“美感”。
- 假装理解：RLHF 鼓励有帮助回答，模型不总是主动承认缺少事实。
- 长上下文失真：摘要损失、旧事实过期、冲突信息并存。
- 后期质量下降：上下文熵增、注意力稀释、错误决策被反复引用。

### 机制拆解

#### Token Prediction

模型不是先构建一个完整工程世界再输出答案，而是在当前上下文下不断预测下一个 token。

工程影响是：当 Prompt 里缺少真实调用链时，模型会补出训练语料中最常见的工程形态。比如看到“实现 memory”，它很容易输出 `MemoryManager`、`VectorStore`、`EmbeddingService`，因为这些词在类似语境里高频出现。

修法不是反复说“不要脑补”，而是提供真实路径：

```text
Memory 必须从 src/context/prepare.ts 注入。
Session 恢复只读 transcript JSONL。
本阶段不实现 vector search。
```

#### Attention

长上下文里，不是每个 token 都同等影响输出。模型会更关注当前任务附近、结构更清晰、和生成目标更相关的信息。

工程影响是：如果关键限制埋在很早的历史里，后面又给了一个相反示例，模型可能跟着后面的示例走。比如早期说“不新增依赖”，后面贴了一个使用新库的代码片段，模型可能认为新增依赖是允许的。

修法是把高优先级约束放到任务附近，并用结构化块标识：

```text
本轮硬约束：
- 不新增依赖。
- 不改 public API。
- 只允许修改 src/context/*。
```

#### Context Window

Context Window 是容量上限，不是理解质量保证。窗口越大，越容易把过期日志、失败探索、旧方案、重复工具结果一起带进去。

工程影响是：模型后期质量下降，经常不是因为窗口不够，而是窗口里噪声太多。大上下文会让模型同时看到“旧方案 A 被否定”和“旧方案 A 的完整代码”，输出时仍可能复活旧方案。

修法是做 Context Projection：本地 transcript 保存完整历史，模型请求只拿当前有效投影视图。

#### RLHF

RLHF 会让模型更倾向给出“有帮助”的回答，而不是频繁拒绝、停下、承认不知道。

工程影响是：当用户说“帮我接上插件系统”但上下文里没有插件入口时，模型可能为了有帮助而发明入口路径。它不是故意撒谎，而是在“给出可用答案”的行为偏好下补全缺失信息。

修法是把“停止条件”写成输出契约：

```text
如果找不到真实入口，必须停止并列出已搜索路径。
不允许假设存在 installPlugin()。
```

#### Sampling

Sampling 会让输出存在随机性。温度越高，模型越可能选择不那么保守的表达和方案。

工程影响是：同一个模糊 Prompt，某次模型可能小改，另一次可能大重构。复杂工程不能依赖“这次刚好听话”。

修法是降低任务自由度，而不是只依赖模型稳定性：

- 明确文件范围。
- 明确不做事项。
- 明确验收命令。
- 明确失败时停止。

#### Instruction Hierarchy

模型会尝试按指令层级行动：系统指令、开发者指令、项目规则、用户任务、上下文示例、历史消息。问题在于工程文档里常常混入互相冲突的规则。

工程影响是：如果长期规则说“不要重构”，当前任务又说“优化架构”，模型可能无法判断到底允许多大改动。它可能选择折中，既改了结构，又没有完整迁移。

修法是显式声明本轮优先级：

```text
本轮优先级：
1. 用户最新限制优先于历史建议。
2. 不改 public API 优先于代码美化。
3. 测试通过优先于抽象统一。
```

#### Tool Feedback

Agent 不是只受 Prompt 影响，也受工具结果影响。工具输出越大、越乱、越像最终事实，模型越可能围绕它继续生成。

工程影响是：一次失败命令如果长期留在上下文里，后面即使已经修复，模型仍可能继续围绕旧错误排查。

修法是把工具结果分级：

- 当前失败：保留完整关键片段。
- 已修复失败：压缩成结论。
- 无关探索：移出主上下文。
- 大输出：写 side file，只注入摘要和引用。

### 最佳实践

- 把最高优先级约束放在靠近任务的位置。
- 用真实文件和测试替代抽象描述。
- 用 Output Contract 逼模型暴露不确定性。
- 定期 Context Reset，不要无限续写。
- 对高风险任务降低自由度，而不是只换更强模型。
- 把“必须停止”的条件写清楚，抵消模型过度 helpful 的倾向。

### 实战经验

不要指望模型“理解你的潜台词”。工程协作里要把潜台词变成显式契约：哪些文件可改、哪些 API 不动、哪些测试必须跑、什么情况下必须停。

### 真实踩坑

在长会话里，早期一句“可以先简化实现”被模型后续反复当成许可，导致后面复杂章节仍然写简化代码。修复方式是阶段刷新 Prompt，明确“从本章开始必须按生产级路径处理，早期简化约束作废”。

## 12. Agentic Workflow

复杂工程不能靠一次 Prompt 完成。

稳定的 Agentic Workflow 会把不同认知任务拆开：

- Planner -> Executor -> Reviewer。
- Architect -> Implementer -> Reviewer。
- Spec -> Plan -> Code -> Test。
- Reflection Loop。
- Self-debugging Workflow。
- Iterative Refinement。

### 错误示例

```text
直接实现这个复杂功能，写完自己检查一下。
```

### 正确示例

```text
工作流：
1. Planner：只读代码，列出影响面、风险、测试策略，不改文件。
2. Executor：按批准计划改最少文件。
3. Reviewer：只读 diff，找 bug、回归、测试缺口。
4. Verifier：运行最小相关命令，给出 PASS/FAIL/PARTIAL。
5. Fix Loop：只修验证失败点。
```

### 为什么错误

同一个上下文里同时设计、实现、自检，模型容易为自己的方案辩护，漏掉假设错误。

### 为什么正确

分角色可以降低上下文污染，让 Reviewer 不继承 Implementer 的乐观偏差。

### 最佳实践

- Planning 阶段禁止改文件。
- Implementation 阶段禁止扩大范围。
- Review 阶段优先找真实 bug，不写总结套话。
- Verification 必须有命令、日志或人工验收证据。
- FAIL 后回到最小修复，不重新设计整个系统。

### 实战经验

Reflection 只适合发现表层问题，不能替代测试。AI 自检最容易说“看起来正确”。真正有效的 Reviewer Prompt 应该要求“尝试打破它”，而不是“确认它正确”。

### 真实踩坑

一个批处理任务中，主 Agent 完成所有 todo 后直接总结，忘了跑测试。后来在 Todo/Task 工具结果里加入“完成 3 个以上任务但无验证时提醒 verification”的门禁，才减少这种提前收尾。

## 13. Context Engineering

Context Engineering 是 Claude Code 类系统最核心的问题之一。

Prompt 决定“模型应该怎么做”，Context 决定“模型实际上能基于什么做”。复杂 Agent 的质量上限，很大程度取决于上下文管线质量。

### 错误示例

```text
把 repo 全量索引结果、最近所有工具输出、完整 transcript、所有 memory 都塞进请求。
```

### 正确示例

```text
Context 管线：
1. 收集候选：项目规则、memory、相关文件、最近消息、工具结果。
2. 过滤过期事实：丢弃已修复错误和废弃方案。
3. 分层预算：system、memory、runtime、messages、tool result 分别限额。
4. 压缩旧历史：保留决策、文件、测试、风险、下一步。
5. 注入当前证据：当前 diff、失败测试、用户最新约束优先。
6. 请求后记录：transcript 保存完整事实，模型输入只保存投影视图。
```

### 为什么错误

全量上下文会熵增。信息越多，冲突越多，模型越难判断什么是当前有效事实。

### 为什么正确

正确管线把上下文当成工程产物：可预算、可压缩、可恢复、可审计。

### 最佳实践

- Context Packing：相关文件、错误日志和任务目标放在相邻位置。
- Context Isolation：子 Agent 只拿子任务需要的上下文。
- Context Refresh：每个阶段结束后生成状态摘要。
- Context Compression：不要只摘要文本，要摘要工程状态。
- Memory Layering：区分用户偏好、项目规则、任务记忆。
- Retrieval Injection：按任务检索，不全量注入。
- Context Window Budgeting：给输出预留空间，避免刚好打满窗口。

### 实战经验

Context 不是越完整越可靠。一个 2,000 token 的高质量任务包，往往比 80,000 token 的混乱历史更能让模型写出好代码。

### 真实踩坑

压缩历史时，如果摘要只写“已经实现权限系统”，后续模型不知道权限入口、数据结构、审计路径。正确摘要必须写“权限判断在 `permissionContext`，结果写 transcript，deny 后 tool_result 返回拒绝原因”。

## 14. AI 代码质量退化

大型工程里的 AI 质量退化是必然问题。

原因不是模型突然变差，而是上下文和代码库都在熵增：局部例外越来越多，历史决策越来越多，命名越来越不一致，Prompt 也越来越厚。

### 错误示例

```text
继续沿着之前的方式写，尽量快。
```

### 正确示例

```text
先做架构回收：
- 列出最近 5 个功能新增的模块。
- 找重复抽象、命名漂移、未接入主链路的代码。
- 只提出 3 个最小 refactor。
- 每个 refactor 必须有测试保护。
```

### 为什么错误

持续“快写”会让模型复制最近代码，不断扩大局部坏味道。

### 为什么正确

正确做法周期性收束架构，让 AI 重新对齐当前系统，而不是继续沿用过期上下文。

### 最佳实践

把质量回收当成固定工程节奏，而不是等系统坏了再补救。每完成几个功能，就让 AI 重新读取当前关键模块，列出重复代码、命名漂移、未接入主链路的抽象和测试缺口，再由人类决定哪些值得修。

### 退化表现

- 越写后面质量越差。
- 架构一致性下降。
- 复制粘贴增多。
- over-engineering 增多。
- 开始用伪代码或 TODO 代替实现。
- 无意义抽象增多。
- 模块边界感下降。

### 解决方式

- 定期 Refactor，但必须小步。
- 架构回收：删除未接入主链路的概念代码。
- Prompt Refresh：重写当前项目规则，丢弃过期约束。
- Context Reset：新会话只带状态摘要和关键文件。
- Codebase Re-index：重新扫描真实代码，而不是依赖记忆。
- 分阶段重构：先加测试，再改结构。
- 重新约束 AI：文件范围、API 稳定性、测试命令。

### 实战经验

AI 后期最容易“看着上下文写代码”，而不是“看着仓库写代码”。解决方式是强制它重新读取相关文件，并在计划里引用当前代码事实。

### 真实踩坑

一个项目连续多轮让 AI 改工具系统，后期它开始复制旧工具的错误 schema。后来新任务统一要求“先打开当前 Tool interface 和两个最近工具实现，再计划”，错误率下降很多。

## 15. Human + AI 协作模式

AI 是高能力工程协作者，不是完全自动工程师。

人类必须负责：

- 业务目标和取舍。
- 架构边界。
- 安全和权限策略。
- 是否引入依赖。
- 是否扩大范围。
- 最终代码 Review。
- 生产风险判断。

AI 更适合负责：

- 阅读和总结代码。
- 生成局部实现。
- 补测试。
- 枚举边界情况。
- 生成迁移方案。
- 做只读 Review。
- 整理文档和复盘。

### 错误示例

```text
你自己决定怎么做，做完告诉我。
```

### 正确示例

```text
你可以自主完成实现细节，但以下事项必须先停下来确认：
- 新增外部依赖。
- 改 public API。
- 删除或移动大量文件。
- 改认证、权限、密钥、网络调用。
- 修改构建和发布流程。
```

### 为什么错误

AI 会把“自己决定”理解成允许架构取舍。它没有真实业务责任，也不知道团队维护成本。

### 为什么正确

正确方式把 AI 的自主权限制在可回滚的工程细节里，把高风险决策留给人类。

### 最佳实践

- 人类定义目标、边界、验收。
- AI 提供方案、影响面、实现和测试。
- 人类批准架构方向。
- AI 执行小步改动。
- AI Reviewer 找问题。
- 人类最终 Review 和合并。

### 实战经验

最好的协作不是“AI 全自动”，而是“人类少做机械工作，多做决策”。当 AI 开始主导架构，你要立刻把任务拉回到具体文件、具体行为、具体验收。

### 真实踩坑

AI 曾建议把权限、审计、策略引擎合成一个“统一治理核心”。概念上漂亮，但会让安全边界和审计职责混在一起。人类决策保留三个层次：权限判断、审计记录、策略匹配，后续扩展反而更稳。

## 交付清单

每次让 AI 做复杂工程任务，都可以用这张清单：

- 目标是否足够小。
- 是否明确不做什么。
- 是否引用真实文件和真实调用链。
- 是否规定允许修改范围。
- 是否有测试或验证命令。
- 是否有输出契约。
- 是否隔离了探索上下文和实现上下文。
- 是否记录了决策摘要。
- 是否需要独立 Review。
- 是否需要人类确认高风险决策。

如果这些问题答不上来，不要急着让 AI 写代码。
